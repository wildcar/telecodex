Ты — senior backend/devops инженер. Нужно разработать Telegram-бота на Python, который управляет Codex CLI через shell (subprocess) на Ubuntu-сервере. Бот поддерживает выбор проекта и сессии, показывает сценарный UI в Telegram и работает как systemd-сервис.

Цели
1) Telegram-бот (python) принимает команды/сообщения.
2) Для каждого чата можно выбрать "проект" (рабочий каталог) и "сессию" (логический контекст/история) без ручного ввода UUID.
3) Запуск Codex CLI выполняется через subprocess в нужном cwd, с управлением окружением и логами.
4) Пользовательский UX строится вокруг одного статусного сообщения, inline-кнопок и компактных карточек состояния.
5) Бот разворачивается как сервис на Ubuntu (systemd), с конфигом через env/.env.

Текущая задача (инкремент)
- Сохранить сценарный интерфейс Telegram поверх существующих команд.
- Во время выполнения задачи показывать не только `typing`, но и периодически обновляемое одно статусное сообщение с понятными этапами работы Codex.
- Формировать промежуточные статусы из human-readable комментариев Codex (`смотрю FS.md`, `проверяю runner.py`), а не стримить сырой stdout/stderr.
- Не показывать в статусах shell-команды, дампы файлов, технические префиксы, длинные фрагменты ответа и прочий шум CLI.
- Финальный ответ по-прежнему должен заменять стартовое статусное сообщение целиком.
- Под финальным ответом показывать action-кнопки: продолжить, новая сессия, сменить проект, лог.
- Сохранить текстовые команды как fallback-интерфейс для админских и редких действий.
- Починить локальный запуск тестов без ручного `PYTHONPATH`.
- Устранить зависание слоя SQLite в текущем окружении при работе через `aiosqlite`.
- При необходимости заменить адаптер доступа к SQLite на более надежный для этого runtime, сохранив асинхронный интерфейс репозитория.
- Закоммитить изменения в git; push выполнить, если среда это позволит.

Принятые решения по умолчанию (для реализации)
- Telegram framework: aiogram v3.
- Хранилище: SQLite; при нестабильности thread-based адаптеров в целевом runtime допустим async-friendly репозиторий с синхронным `sqlite3` внутри коротких операций.
- Конфиг проектов: env-переменная `TELECODEX_PROJECTS_JSON` с JSON-объектом формата:
  `{"demo":"/opt/projects/demo","infra":"/opt/projects/infra"}`.
- Текущий выбор проекта/сессии хранится по `chat_id` (не по `user_id`).
- Команда запуска Codex задается переменной `CODEX_COMMAND`.

Ограничения и допущения
- Без внешних БД на старте: использовать локальный файл SQLite.
- Должна быть возможность работать с несколькими проектами (несколько рабочих каталогов).
- Одновременно в одном чате не должно запускаться больше 1 активной задачи (блокировка).
- Безопасность: whitelist каталогов проектов, запрет произвольных shell-команд от пользователя, ограничение прав пользователя сервиса.
- В пользовательском UI не показывать filesystem paths и полные UUID, кроме fallback-команд и технических логов.

Функциональные требования (Telegram UX)
A) Главное меню
- `/start` и `/menu` показывают карточку состояния текущего чата.
- Карточка содержит: выбранный проект, выбранную сессию (alias или короткий id), статус (`idle`/`running`), последнюю активность.
- Под карточкой inline-кнопки: `Проект`, `Сессия`, `Новая сессия`, `История`, `Стоп`, `Помощь`.

B) Управление проектами
- `/projects` — показать доступные проекты компактным списком без путей.
- `/project <name>` — fallback-выбор проекта для текущего чата.
- Inline-кнопка `Проект` открывает список доступных проектов.
- При смене проекта текущая выбранная сессия сбрасывается.

C) Управление сессиями
- `/sessions` — показать список сессий для текущего проекта в компактном виде: alias или короткий id, дата обновления.
- `/session <id|new>` — fallback: выбрать существующую сессию или создать новую.
- `/session_name <alias>` — назначить имя текущей сессии.
- `/whereami` — вывести ту же карточку состояния, что и `/menu`, но без избыточной технической информации.
- Inline-кнопка `Сессия` открывает список последних сессий текущего проекта.
- Inline-кнопка `Новая сессия` создает и выбирает новую сессию.

Модель сессии
- session_id (uuid)
- project_name / project_path
- created_at, updated_at
- alias (optional)
- history_log_path
- codex_resume_ref (optional)

D) Запуск задач
- Любой текст без команды — задача для Codex.
- `/run <text>` — явный fallback-запуск.
- `/status` — компактный статус активной задачи.
- `/cancel` — отмена текущего процесса (terminate/kill).
- `/last` — последние X строк лога по текущей сессии.

E) Вывод в Telegram
1) Бот отправляет одно стартовое сообщение `Running...`.
2) Пока Codex выполняется, бот периодически отправляет `typing`.
3) Пока Codex выполняется, бот периодически редактирует то же стартовое сообщение, показывая короткие понятные статусы текущей работы и elapsed time.
4) Эти статусы извлекаются только из осмысленных комментариев Codex; сырые чанки stdout/stderr, shell-команды, file dumps и длинные блоки текста в чат не публикуются.
5) В конце стартовое сообщение заменяется финальным очищенным ответом.
6) Под финальным ответом добавляются inline-кнопки: `Продолжить`, `Новая сессия`, `Сменить проект`, `Лог`.
7) Полный технический вывод сохранять только в лог сессии; при большом объеме можно отправлять как документ.

Технические требования (Python)
- Асинхронная архитектура.
- Управление процессами: `asyncio.create_subprocess_exec`.
- Логи: `logging` + `RotatingFileHandler`.
- Конфиг: `.env` + `pydantic-settings`.
- Тесты должны запускаться в локальном `.venv` командой вида `./.venv/bin/pytest -q` без обязательной ручной установки `PYTHONPATH`.

Интеграция с Codex CLI
- Бот не выполняет произвольные команды пользователя.
- Запускается только `CODEX_COMMAND` с добавлением prompt.
- Перед запуском выставляется `cwd = project_path`.
- История хранится в БД и подмешивается в prompt (последние N сообщений).
- Пользовательский запрос текущего запуска не должен дублироваться в сохраненной истории до завершения ответа.
