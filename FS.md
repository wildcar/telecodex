Ты — senior backend/devops инженер. Нужно разработать Telegram-бота на Python, который управляет Codex CLI через shell (subprocess) на Ubuntu-сервере. Бот поддерживает выбор проекта и сессии, показывает сценарный UI в Telegram и работает как systemd-сервис.

Цели
1) Telegram-бот (python) принимает команды/сообщения.
2) Для каждого чата можно выбрать проект и работать с сессиями Codex без ручного ввода UUID в обычном UX.
3) Запуск Codex CLI выполняется через subprocess в нужном cwd, с управлением окружением и логами.
4) Пользовательский UX строится вокруг одного статусного сообщения, inline-кнопок и компактных карточек состояния.
5) Бот разворачивается как сервис на Ubuntu (systemd), с конфигом через env/.env.

Текущая задача (инкремент)
- Сохранить сценарный интерфейс Telegram поверх существующих команд.
- Добавить системное меню команд Telegram (`setMyCommands`), чтобы основные команды были видны в UI бота без ручного ввода.
- Вести только один вид пользовательского текстового лога истории: plaintext-файл `conversation<TelegramUserID>.log`.
- В conversation-log сохранять для каждого запуска: timestamp, исходное сообщение пользователя, полную shell-команду запуска Codex и полный сырой ответ Codex без форматирования и фильтрации.
- Полностью убрать локальное хранение истории диалога в БД и любые отдельные history-log файлы с сырым выводом по сессии.
- Бот должен опираться только на session id, возвращаемый самим Codex CLI; внутренние UUID сессий бота запрещены.
- Выбор текущей сессии в чате хранить как выбранный `codex_session_id` или `null`, если пользователь запросил новую сессию, но еще не выполнил первый запуск.
- Упростить промежуточный Telegram-статус: убрать служебные префиксы вроде `Сейчас:` и `Недавно:`, показывать только короткие содержательные апдейты хода работы.
- Гарантировать доставку финального ответа в Telegram даже если ответ Codex превышает лимит длины сообщения: стартовое сообщение должно завершаться валидным финальным текстом, а полный ответ при необходимости отправляться отдельным файлом.
- Проверить и устранить ошибки `MESSAGE_TOO_LONG` / `text is too long` в runtime-логах бота.
- Во время выполнения задачи показывать не только `typing`, но и периодически обновляемое одно статусное сообщение с понятными этапами работы Codex.
- Формировать промежуточные статусы из human-readable комментариев Codex (`смотрю FS.md`, `проверяю runner.py`), а не стримить сырой stdout/stderr.
- Не показывать в статусах shell-команды, дампы файлов, технические префиксы, длинные фрагменты ответа и прочий шум CLI.

- Под финальным ответом показывать action-кнопки: новая сессия, сменить проект.
- Сохранить текстовые команды как fallback-интерфейс для админских и редких действий.

- Ограничить `/restart` whitelist-списком admin chat id из env-конфига.
- Не выполнять `/restart`, если в любом чате есть активный запуск Codex; сначала требуется явная отмена/дожидание завершения.
- Починить локальный запуск тестов без ручного `PYTHONPATH`.
- Устранить зависание слоя SQLite в текущем окружении при работе через `aiosqlite`.
- При необходимости заменить адаптер доступа к SQLite на более надежный для этого runtime, сохранив асинхронный интерфейс репозитория.
- Закоммитить изменения в git и отправить их в GitHub.

Принятые решения по умолчанию (для реализации)
- Telegram framework: aiogram v3.
- Хранилище: SQLite; при нестабильности thread-based адаптеров в целевом runtime допустим async-friendly репозиторий с синхронным `sqlite3` внутри коротких операций.
- Конфиг проектов: env-переменная `TELECODEX_PROJECTS_JSON` с JSON-объектом формата:
  `{"demo":"/opt/projects/demo","infra":"/opt/projects/infra"}`.
- Текущий выбор проекта/сессии хранится по `chat_id` (не по `user_id`).
- Команда запуска Codex задается переменной `CODEX_COMMAND`; по умолчанию бот запускает Codex без sandbox/approval-ограничений.

Ограничения и допущения
- Без внешних БД на старте: использовать локальный файл SQLite.
- Должна быть возможность работать с несколькими проектами (несколько рабочих каталогов).
- Одновременно в одном чате не должно запускаться больше 1 активной задачи (блокировка).

Функциональные требования (Telegram UX)
A) Главное меню
- `/start` и `/menu` показывают карточку состояния текущего чата.
- При старте приложения бот публикует Telegram command menu с краткими описаниями основных пользовательских и fallback-команд.
- Карточка содержит: выбранный проект, project path, выбранную сессию (alias или полный codex session id), статус (`idle`/`running`), последнюю активность.
- Под карточкой inline-кнопки: `Проект`, `Сессия`, `Новая сессия`, `Стоп`, `Помощь`.

B) Управление проектами
- `/projects` — показать доступные проекты списком с путями.
- `/project <name>` — fallback-выбор проекта для текущего чата.
- Inline-кнопка `Проект` открывает список доступных проектов.
- При смене проекта текущая выбранная сессия сбрасывается.

C) Управление сессиями
- `/sessions` — показать список сохраненных Codex-сессий для текущего проекта: alias или полный codex session id, дата обновления.
- `/session new` — сбросить текущую выбранную сессию; следующий запуск должен стартовать новую сессию Codex.
- `/session <codex_session_id>` — fallback: выбрать ранее сохраненную Codex-сессию текущего проекта.
- `/session_name <alias>` — назначить имя текущей Codex-сессии.
- `/whereami` — вывести ту же карточку состояния, что и `/menu`, но без избыточной технической информации.
- Inline-кнопка `Сессия` открывает список последних Codex-сессий текущего проекта.
- Inline-кнопка `Новая сессия` не создает локальную запись заранее, а только сбрасывает текущую выбранную Codex-сессию.

Модель сессии
- codex_session_id
- project_name / project_path
- created_at, updated_at
- alias (optional)

D) Запуск задач
- Любой текст без команды — задача для Codex.
- `/run <text>` — явный fallback-запуск.
- `/status` — компактный статус активной задачи.
- `/cancel` — отмена текущего процесса (terminate/kill).

E) Вывод в Telegram
1) Бот отправляет одно стартовое сообщение `Running...`.
2) Пока Codex выполняется, бот периодически отправляет `typing`.
3) Пока Codex выполняется, бот периодически редактирует то же стартовое сообщение, показывая короткие понятные статусы текущей работы и elapsed time.
4) Эти статусы извлекаются только из осмысленных комментариев Codex; сырые чанки stdout/stderr, shell-команды, file dumps и длинные блоки текста в чат не публикуются.
5) В конце стартовое сообщение заменяется финальным очищенным ответом.
6) Под финальным ответом добавляются inline-кнопки: `Новая сессия`, `Сменить проект`.
7) Технический вывод сохранять только во внутренних логах сервиса и в conversation-log; в пользовательский Telegram UX он не выводится.
8) Если запуск стартовал без выбранной сессии, бот выполняет обычный `codex exec <prompt>`, извлекает из вывода `session id:` и сохраняет эту Codex-сессию для следующих сообщений.
9) Если для чата уже выбрана Codex-сессия, бот продолжает работу только через `codex exec resume <codex_session_id> <prompt>`.

F) Админ-команды сервиса
- `/restart` доступна только chat id, перечисленным в `TELECODEX_ADMIN_CHAT_IDS`.
- При вызове `/restart` бот отвечает подтверждением в Telegram и после короткой задержки инициирует штатную остановку polling-процесса.
- После остановки процесс завершается, а `systemd` перезапускает сервис согласно unit-конфигурации.
- Если есть хотя бы один активный запуск Codex, `/restart` возвращает отказ без остановки процесса.

Технические требования (Python)
- Асинхронная архитектура.
- Управление процессами: `asyncio.create_subprocess_exec`.
- Логи: `logging` + `RotatingFileHandler`.
- Conversation-log писать в обычный текстовый файл без markdown/очистки, с сохранением исходных переносов строк.
- Конфиг: `.env` + `pydantic-settings`.
- Тесты должны запускаться локально командой `cd /home/codex/telecodex_bot && ./.venv/bin/pytest -q` без обязательной ручной установки `PYTHONPATH`.

Интеграция с Codex CLI
- Бот не выполняет произвольные команды пользователя.
- Запускается только `CODEX_COMMAND` с добавлением prompt.
- Перед запуском выставляется `cwd = project_path`.
- Запуск из бота не должен ограничивать сетевой доступ Codex sandbox-настройками по умолчанию.
- Для продолжения диалога использовать только родной механизм сессий Codex через `codex exec resume <codex_session_id> <prompt>`.
- Локальная история сообщений и отдельные session history-логи не используются и не сохраняются.
- Источник истины для идентификатора сессии — только `session id`, возвращаемый самим Codex CLI.
