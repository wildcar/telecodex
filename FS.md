Ты — senior backend/devops инженер. Нужно разработать Telegram-бота на Python, который управляет Codex CLI через shell (subprocess) на Ubuntu-сервере. Бот поддерживает выбор проекта и сессии, стриминг вывода в Telegram, и работает как systemd-сервис.

Цели
1) Telegram-бот (python) принимает команды/сообщения.
2) Для каждого чата можно выбрать "проект" (рабочий каталог) и "сессию" (логический контекст/история) для Codex.
3) Запуск Codex CLI выполняется через subprocess в нужном cwd, с управлением окружением и логами.
4) Вывод Codex показывается в Telegram потоком (частичные обновления сообщения), без спама сотнями сообщений.
5) Бот разворачивается как сервис на Ubuntu (systemd), с конфигом через env/.env.

Текущая задача (инкремент)
- Убрать стриминг промежуточного вывода из пользовательского чата.
- Во время выполнения бот должен показывать одно стартовое сообщение `Running...` и периодически отправлять chat action `typing`, пока ответ не готов.
- После завершения бот должен заменить стартовое сообщение одним финальным очищенным ответом без дополнительных служебных сообщений.
- Убрать дублирование пользовательского текста в начале ответа и подавить повторяющиеся фрагменты, возникающие из-за эха prompt/истории.
- Не подмешивать текущий пользовательский запрос в `recent_history` перед запуском Codex.
- Очищать и сворачивать сохраненную историю перед сборкой prompt, чтобы исключить многострочное эхо старых ответов.
- Закоммитить и отправить изменения в GitHub.

Принятые решения по умолчанию (для реализации)
- Telegram framework: aiogram v3.
- Хранилище: SQLite + aiosqlite.
- Конфиг проектов: env-переменная `TELECODEX_PROJECTS_JSON` с JSON-объектом формата:
  `{"demo":"/opt/projects/demo","infra":"/opt/projects/infra"}`.
- Текущий выбор проекта/сессии хранится по `chat_id` (не по user_id).
- Команда запуска Codex задается переменной `CODEX_COMMAND`.

Ограничения и допущения
- Без внешних БД на старте: использовать локальный файл SQLite.
- Должна быть возможность работать с несколькими проектами (несколько рабочих каталогов).
- Одновременно в одном чате не должно запускаться больше 1 активной задачи (блокировка).
- Безопасность: whitelist каталогов проектов, запрет произвольных shell-команд от пользователя, ограничение прав пользователя сервиса.

Функциональные требования (Telegram команды)
A) Управление проектами
- /projects — показать доступные проекты (имя -> путь).
- /project <name> — выбрать проект для текущего чата.
- /pwd — показать текущий выбранный проект (cwd).
- Разрешены только whitelisted пути из `TELECODEX_PROJECTS_JSON`.

B) Управление сессиями
- /sessions — показать список сессий для текущего проекта (последние N).
- /session <id|new> — выбрать существующую сессию или создать новую.
- /session_name <alias> — назначить имя текущей сессии.
- /whereami — вывести: выбранный проект, выбранная сессия (id), alias.

Модель сессии
- session_id (uuid)
- project_name / project_path
- created_at, updated_at
- alias (optional)
- history_log_path
- codex_resume_ref (optional)

C) Запуск задач
- Любой текст без команды — задача для Codex (prompt).
- /run <text> — явный запуск.
- /status — статус активной задачи: идет/нет, сколько времени, текущий проект/сессия.
- /cancel — отмена текущего процесса (terminate/kill).
- /last — последние X строк лога по текущей сессии.

D) Вывод в Telegram
1) Бот отправляет стартовое сообщение `Running...`.
2) Пока Codex выполняется, бот периодически отправляет chat action `typing`.
3) Промежуточные чанки stdout/stderr в пользовательский чат не публикуются.
4) В конце стартовое сообщение заменяется финальным очищенным ответом.
5) Полный технический вывод сохранять только в лог сессии; при большом объеме можно отправлять как документ по служебной логике, но не как чат-стрим.

Технические требования (Python)
- Асинхронная архитектура.
- Управление процессами: asyncio.create_subprocess_exec.
- Логи: logging + RotatingFileHandler.
- Конфиг: .env + pydantic-settings.

Интеграция с Codex CLI
- Бот не выполняет произвольные команды пользователя.
- Запускается только `CODEX_COMMAND` с добавлением prompt.
- Перед запуском выставляется cwd = выбранный project_path.
- История хранится в БД и подмешивается в prompt (последние N сообщений).
